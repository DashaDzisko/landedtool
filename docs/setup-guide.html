Что мы вообще будем делать

Представь, что кто-то приготовил блюдо и записал рецепт. Проект — это и есть «рецепт» приложения: набор файлов с кодом. Сам по себе он не работает — его нужно «приготовить» на своём компьютере.

Наш план простой:

  1. Поставить Cursor — это программа-«кухня», в которой мы открываем код.
  2. Поставить Node.js — это «плита», без которой код не запустится.
  3. Скачать сам проект и распаковать его.
  4. Запустить его — и приложение откроется прямо в твоём браузере.

Идём по порядку и ничего не пропускаем. Каждый шаг занимает пару минут.

1 Устанавливаем Cursor

Cursor — это редактор кода со встроенным ИИ-помощником. По сути это программа, где мы будем смотреть и менять файлы проекта, а ещё запускать его.

  1. Зайди на сайт cursor.com.
  2. Нажми большую кнопку Download — сайт сам поймёт, какая у тебя система (Mac или Windows).
  3. Открой скачанный файл и установи программу, как любое другое приложение:
    • Mac: перетащи иконку Cursor в папку Applications.
    • Windows: запусти установщик .exe и жми «Далее» до конца.
  4. Запусти Cursor. При первом запуске он спросит про вход и настройки — можно везде нажимать «Continue» / «Skip», для нас сейчас это не важно.
💡 Подсказка: Cursor сделан на основе очень популярного редактора VS Code, поэтому почти всё в нём выглядит и работает так же.

2 Устанавливаем Node.js

Наш проект написан на JavaScript. Чтобы компьютер умел его «готовить», нужна Node.js — это та самая «плита», которая запускает код. Вместе с ней автоматически ставится npm — помощник, который скачивает все нужные проекту детали.

💡 Самый простой способ — попросить ИИ в Cursor. Открой чат Cursor (значок чата справа или Cmd/Ctrl + L) и напиши прямо человеческими словами: 
Установи мне Node.js LTS на этот компьютер и проверь, что node и npm работают
Cursor сам предложит команды и выполнит их за тебя — просто соглашайся (Run / Accept), когда он спрашивает. Это удобнее, чем ставить вручную.

Способ вручную (если хочешь сделать сам)

Вариант А — скачать с сайта (самый понятный):

  1. Зайди на nodejs.org.
  2. Скачай версию с пометкой LTS (это «стабильная, проверенная» версия — то, что нам нужно).
  3. Открой скачанный файл и установи, нажимая «Далее» / «Continue» до конца.

Вариант Б — одной командой в терминале Cursor (терминал открывается так: Terminal → New Terminal):

  • Mac: brew install node  (если установлен Homebrew)
  • Windows: winget install OpenJS.NodeJS.LTS

Проверка, что всё получилось. Открой в Cursor терминал и набери:

node --version

Если в ответ появилось что-то вроде v20.11.0 — отлично, Node.js на месте.

ℹ️ Что такое терминал? Это окошко, куда команды печатают текстом, а не мышкой. Все команды из этой инструкции можно просто копировать и вставлять.

3 Скачиваем репозиторий

Репозиторий (или просто «репо») — это папка с проектом, которая лежит в интернете, обычно на сайте GitHub. Наша задача — скачать её копию к себе.

🔗 Ссылка на проект: github.com/DashaDzisko/landedtool
  1. Открой ссылку на репозиторий выше.
  2. Найди зелёную кнопку <> Code вверху справа.
  3. В выпавшем меню нажми Download ZIP.
  4. Файл-архив .zip скачается в твою папку Загрузки / Downloads.
ℹ️ Что такое ZIP? Это «сжатая» папка — много файлов, упакованных в один, чтобы их было удобно передавать. Перед использованием её нужно «распаковать» (распаковать = unzip).

4 Распаковываем папку

  • Mac: просто дважды кликни по файлу .zip — рядом появится обычная папка.
  • Windows: кликни по файлу правой кнопкой → Извлечь всё / Extract All → Извлечь.
💡 Совет: перенеси распакованную папку в удобное место, например на Рабочий стол, чтобы её было легко найти. И запомни путь к ней — он скоро понадобится.

5 Открываем проект в Cursor

  1. Открой Cursor.
  2. В верхнем меню выбери File → Open Folder… (Файл → Открыть папку).
  3. Найди распакованную папку проекта и нажми Open.
  4. Если Cursor спросит «Do you trust the authors?» («Доверяешь авторам?») — нажми Yes, I trust.

Слева появится список файлов проекта. Это и есть «рецепт». Файлов много, но напрямую нам понадобятся всего пара из них.

6 Устанавливаем зависимости

Зависимости — это готовые детали (чужой код), которые проект использует, чтобы не писать всё с нуля. Их нужно скачать один раз.

Сначала открой терминал прямо внутри Cursor:

  • В верхнем меню: Terminal → New Terminal (Терминал → Новый терминал).
  • Или нажми Ctrl + ` (значок слева от цифры 1).

Внизу откроется то самое «чёрное окошко». Важно: оно уже открыто внутри папки проекта, так что просто впиши команду и нажми Enter:

npm install

Пойдёт скачивание — это может занять 1–3 минуты. Появится много текста, иногда жёлтые предупреждения (warnings) — это нормально, их можно игнорировать. Когда снова появится мигающий курсор и можно печатать — значит, готово.

ℹ️ После этой команды в проекте появится папка node_modules — там и лежат все скачанные детали. Её не нужно открывать или трогать.

7 Создаём базу данных в Supabase

ℹ️ У каждого будет своя база. Ты создашь свой собственный проект Supabase, поэтому твои URL и ключ будут уникальными — не такими, как у других студентов. Это нормально: на этом шаге как раз и нужно получить свои значения и вписать их в .env.local (шаг 9).

Сейчас у нас есть код приложения, но данные ему пока негде хранить. Создадим бесплатную базу данных в Supabase и зальём в неё нашу «схему» — таблицы для заявок, чатов, профилей. Все нужные SQL-файлы уже лежат в проекте, в папке supabase/migrations — нам останется их просто скопировать и запустить.

7.1. Регистрируемся и создаём проект

  1. Зайди на supabase.com и нажми Start your project. Войти проще всего через свой аккаунт GitHub.
  2. Нажми New project (Новый проект).
  3. Заполни:
    • Name — любое имя, например landed.
    • Database Password — придумай пароль для базы. Запиши его где-нибудь — он ещё может пригодиться.
    • Region — выбери регион поближе к себе (например Europe).
  4. Нажми Create new project и подожди 1–2 минуты, пока Supabase всё подготовит.

7.2. Запускаем SQL-файлы (создаём таблицы)

В нашем проекте в папке supabase/migrations лежат три файла, и их нужно запустить строго по порядку:

  1. 0001_init.sql — создаёт сами таблицы (заявки, чаты, профили).
  2. 0002_rls.sql — включает защиту: каждый пользователь видит только свои данные.
  3. 0003_storage.sql — место для хранения файлов (загруженные CV).

Для каждого файла по очереди делаем одно и то же:

  1. В Supabase слева открой раздел SQL Editor (иконка с надписью SQL).
  2. Нажми + New query (Новый запрос).
  3. В Cursor открой файл supabase/migrations/0001_init.sql, выдели весь текст (Cmd/Ctrl + A) и скопируй (Cmd/Ctrl + C).
  4. Вставь его в окно SQL Editor в Supabase и нажми Run (или Cmd/Ctrl + Enter).
  5. Внизу должно появиться зелёное Success. Готово — повтори то же самое для 0002_rls.sql, а затем для 0003_storage.sql.
⚠️ Важно: порядок имеет значение — сначала 0001, потом 0002, потом 0003. Если запустить не по порядку, будут ошибки, потому что второй файл опирается на таблицы из первого.

7.3. Забираем «адрес» и «ключ» базы

Теперь нужно сказать нашему приложению, где живёт база. Эти два значения мы позже впишем в файл .env.local (шаг 9).

На главной странице проекта в Supabase вверху есть кнопка Connect (рядом с названием проекта). Нажми её — откроется выпадающее меню, и там сразу всё нужное:

  1. Project URL — это NEXT_PUBLIC_SUPABASE_URL (выглядит как https://что-то.supabase.co). Нажми Copy рядом.
  2. Publishable key — это NEXT_PUBLIC_SUPABASE_ANON_KEY (начинается с sb_publishable_…). Тоже скопируй.
  3. Сохрани оба значения в заметке — пригодятся через пару шагов.
ℹ️ В этом же меню есть «Direct connection string» и «CLI setup commands» — их трогать не нужно. Нам хватит двух значений сверху. Publishable key (раньше он назывался «anon key») — публичный, он специально попадает в браузер, и это нормально. Реальную защиту данных делает «RLS» из файла 0002_rls.sql: даже с этим ключом человек видит только свои записи. А вот secret / service_role ключ (если встретишь) — секретный, в браузер его выкладывать нельзя.
✅ Не нужно входить в Supabase через терминал. Ты можешь встретить команды вроде supabase login или supabase init (в дропдауне «CLI setup commands»). Нам они не нужны. Ключи начинают работать сами, как только ты впишешь Project URL и Publishable key в файл .env.local (шаг 9). Терминальный вход — это отдельный, более продвинутый способ управлять базой, и для запуска нашего проекта он не требуется.

8 Получаем ключ OpenAI

Этот ключ нужен только для ИИ-чата внутри приложения. Если сейчас хочешь просто запустить проект и посмотреть интерфейс — этот шаг можно пропустить и вернуться позже.

  1. Зайди на platform.openai.com/api-keys и войди (или зарегистрируйся).
  2. Нажми Create new secret key, дай ей любое имя.
  3. Скопируй ключ сразу — он показывается только один раз. Это значение OPENAI_API_KEY.
⚠️ Важно: ключ OpenAI — секретный и привязан к деньгам/лимитам аккаунта. Никому его не показывай и не выкладывай в интернет. Скорее всего, его выдаст преподаватель — тогда отдельный аккаунт создавать не нужно.

9 Добавляем секретные ключи

Теперь соберём вместе ключи, которые мы получили в шагах 7 и 8. Эти ключи — как пароли, поэтому они не хранятся прямо в коде, а лежат в отдельном секретном файле .env.local.

  1. В списке файлов слева найди файл .env.example — это образец.
  2. Создай рядом новый файл с именем .env.local (точка в начале обязательна).
  3. Скопируй в него содержимое .env.example и подставь свои настоящие значения:
NEXT_PUBLIC_SUPABASE_URL=https://твой-проект.supabase.co   # Project URL, шаг 7.3
NEXT_PUBLIC_SUPABASE_ANON_KEY=sb_publishable_…             # Publishable key, шаг 7.3
OPENAI_API_KEY=твой-ключ-openai                          # из шага 8
⚠️ Важно: файл .env.local и ключи внутри — секретные. Не выкладывай их в интернет и не отправляй в общий чат. Если хочешь пока запустить проект без ИИ-чата — строку OPENAI_API_KEY можно временно оставить пустой.

10 Запускаем локальный сервер

Это финальный шаг! В терминале (там же, где делали npm install) набери:

npm run dev

Через несколько секунд появится сообщение примерно такое:

  ▲ Next.js
  - Local:   http://localhost:3000
  ✓ Ready in 2.1s

Теперь:

  1. Открой браузер (Chrome, Safari — любой).
  2. В адресной строке набери: http://localhost:3000
  3. Нажми Enter — приложение откроется.
💡 Совет: в Cursor можно навести курсор на ссылку http://localhost:3000 прямо в терминале, зажать Cmd (Mac) или Ctrl (Windows) и кликнуть — браузер откроется сам.

Вход в приложение (Supabase попросит авторизоваться)

Важно: это приложение закрыто за входом. Когда ты откроешь localhost:3000, тебя сразу перебросит на страницу Sign in — это нормально, так и задумано. Вход устроен без пароля, по «волшебной ссылке» (magic link) через Supabase:

  1. Введи свой email и нажми отправить ссылку.
  2. Появится сообщение «Проверь почту» — открой свой почтовый ящик.
  3. Найди письмо от Supabase и нажми ссылку внутри. Тебя автоматически вернёт в приложение уже авторизованным.
ℹ️ Письмо не пришло? Проверь папку «Спам». Учти: на бесплатном Supabase письма иногда приходят с задержкой и есть лимит на их количество в час. Письма со входами видно и в самом Supabase: раздел Authentication → Users (там же появляются все, кто вошёл).
⚠️ Ссылка из письма «уводит не туда» или вход не срабатывает? Значит, Supabase не знает про адрес localhost:3000. Открой в Supabase Authentication → URL Configuration и проверь, что:
  • Site URL = http://localhost:3000
  • в Redirect URLs добавлен http://localhost:3000/**
У новых проектов это обычно уже стоит по умолчанию — но если вход не работает, проверь здесь в первую очередь.
ℹ️ Как остановить сервер? Вернись в терминал и нажми Ctrl + C. Чтобы запустить снова — опять npm run dev. Пока сервер работает, это окно терминала должно оставаться открытым.

Словарик: что это за страшные слова

🖥️

Локальный сервер

«Локальный» значит «на твоём компьютере». Обычно сайты живут где-то в интернете на чужих серверах. Когда мы запускаем npm run dev, твой собственный компьютер на время превращается в маленький сервер и показывает сайт только тебе по адресу localhost:3000.

localhost = «этот самый компьютер». 3000 = номер «двери» (порта), через которую отдаётся сайт. Никто из интернета его не видит — это твоя личная песочница для разработки.

📦

Локальное хранилище (Local Storage)

Это маленькая «записная книжка» прямо внутри браузера. Сайт может сохранять туда небольшие данные — например, что ты выбрал тёмную тему или не закрыл какое-то окно. Эти данные хранятся на твоём устройстве, в твоём браузере, и остаются даже после перезагрузки страницы.

Важное отличие от базы данных: local storage привязан к одному браузеру на одном компьютере. Откроешь сайт с телефона — там этих данных уже не будет. Для серьёзных данных (аккаунты, заявки) используют базу данных в облаке — например Supabase.

🗄️

Supabase (база данных в облаке)

База данных — это умная таблица в интернете, где хранятся все данные приложения: пользователи, их заявки на работу, заметки и т.д. В отличие от local storage, эти данные живут в облаке и доступны с любого устройства.

Supabase — это готовый сервис, который даёт нам такую базу данных «под ключ», а ещё умеет:

  • хранить данные (на основе классической базы PostgreSQL);
  • заниматься входом в аккаунт — в нашем проекте вход по «магической ссылке»: ты вводишь email, тебе приходит письмо, кликаешь — и ты внутри, без пароля;
  • защищать данные, чтобы каждый пользователь видел только своё.

Те самые ключи SUPABASE_URL и SUPABASE_ANON_KEY из шага 7 — это «адрес» и «пропуск» к нашей базе.

🤖

OpenAI (ИИ-чат)

В приложении есть чат с ИИ-помощником. За «мозги» этого чата отвечает OpenAI — та же компания, что сделала ChatGPT. Ключ OPENAI_API_KEY позволяет нашему приложению обращаться к их ИИ. Этот ключ работает «за кулисами» (на сервере) и никогда не показывается в браузере.

🔑

API-ключ

Сначала про API. Представь ресторан: ты (приложение) не идёшь сам на кухню, а говоришь заказ официанту, и он приносит блюдо. API — это и есть такой «официант»: способ, которым одна программа просит что-то у другой (например, наше приложение просит данные у Supabase или ответ у ИИ от OpenAI).

API-ключ — это твой «именной пропуск» к такому сервису. По нему сервис понимает, кто обращается, и считает использование (а иногда и деньги). Именно поэтому ключи OPENAI_API_KEY и ключ Supabase — секретные: если их узнает чужой, он сможет пользоваться сервисом от твоего имени. Поэтому мы и прячем их в файл .env.local, а не пишем прямо в коде.

📡

GET и POST запросы

Когда приложение общается с сервером через API, оно отправляет запросы (requests). У запроса есть «тип», который говорит, что мы хотим сделать. Два самых частых:

  • GET — «дай посмотреть». Это запрос на получение данных, он ничего не меняет. Например: «покажи список моих заявок на работу». Когда ты просто открываешь страницу в браузере — это GET.
  • POST — «прими и сохрани». Это запрос на создание / отправку данных. Например: «сохрани новую заявку» или «отправь это сообщение в чат». POST что-то меняет на сервере.

Простая аналогия с тем же официантом: GET — «принесите, пожалуйста, меню» (только смотрю), POST — «запишите мой заказ» (что-то добавилось). Есть и другие типы (PUT/PATCH — изменить, DELETE — удалить), но для старта достаточно знать эти два.

⚙️

Ещё пара слов на будущее

  • npm — помощник, который скачивает чужой код (зависимости) и запускает проект командами вроде npm run dev.
  • Next.js — «каркас» (фреймворк), на котором построено наше приложение. Он же поднимает локальный сервер.
  • Терминал / командная строка — окно, где мы общаемся с компьютером текстовыми командами.
  • Браузер — программа для просмотра сайтов (Chrome, Safari, Firefox). Именно в нём мы видим результат.

Если что-то пошло не так

command not found: npm или node

Node.js не установился или Cursor его пока «не увидел». Вернись к шагу 2, переустанови Node.js, затем полностью закрой и снова открой Cursor и попробуй ещё раз.

Порт 3000 занят (port 3000 is already in use)

Значит, сервер уже где-то запущен. Либо открой просто http://localhost:3000 в браузере, либо останови старый процесс (Ctrl + C в том терминале), либо Next.js сам предложит порт 3001 — тогда открывай его.

Страница белая или ошибка про Supabase / переменные окружения

Чаще всего не заполнен файл .env.local (шаг 9) или в нём опечатка. Проверь, что имя файла именно .env.local (с точкой в начале) и значения скопированы без лишних пробелов. После изменения .env.local нужно остановить сервер (Ctrl+C) и запустить заново npm run dev.

Команда «висит» и ничего не происходит

Для npm install это нормально — подожди до 2–3 минут. Если завис npm run dev, но в нём написано Ready — всё хорошо, так и должно быть, просто открывай браузер. Сервер «висит» специально: он работает, пока окно открыто.

Совсем застрял(а)?

Порядок действий, который выручает в большинстве случаев:

  1. Спроси у ИИ-чата. Скопируй весь текст ошибки из терминала и вставь в чат Cursor (Cmd/Ctrl + L) — напиши «у меня вот такая ошибка, помоги исправить». Чаще всего он подсказывает, что делать дальше.
  2. Пересмотри лекцию или эту инструкцию. Вернись к нужному шагу и перечитай — часто ответ уже там.
  3. Спроси в учебном чате или у преподавателя. Чтобы помогли быстрее — приложи скриншот ошибки и напиши, на каком шаге застрял(а).
Если приложение открылось в браузере — всё получилось, проект запущен.
